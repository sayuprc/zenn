## はじめに

個人開発でフロントエンドに Astro を採用しています。
Cloud Build 上で SSG ビルドを行い、Cloudflare Workers にデプロイするという構成です。

データの取得は、当初次のような流れになっていました。

1. 概要データを全件取得する
2. 詳細データを 1 件ずつ取得する(並列)

データの件数が増えたとき、この構成のままビルドができなくなったので、見直したときの記録です。

## 課題

詳細データを件数分だけ個別に取得しており、さらに 1 API リクエストの裏で 6 件ほどのクエリが走っていました。
そのため、ページ数 × 6 のクエリがビルド時に DB へ集中していました。
その結果、Cloud Build 上では DB 側が負荷に耐えきれず、ビルドは一度も完走できませんでした。

## 対策

### 1. API の N+1 解消

これまでは「概要データを全件取得」してから「それぞれの詳細データを取得」という流れでした。
これを、概要データを取得する API のレスポンスに詳細データも含める形へ変更し、API リクエストを 1 回にまとめました。

これができたのは、概要データと詳細データの差が小さく、概要に相乗りさせてもレスポンスが大きくなりすぎなかったためです。
詳細データが大きくなると、今度はレスポンスの肥大化が問題になります。
その場合は、詳細データを分けたうえで、次のような対策が必要になります。

- クエリの見直し
- `p-limit` などによる同時実行制御
- `updatedAt` などを用いた増分取得

### 2. Content Collections の活用

Content Collections は、Markdown・JSON・YAML・外部 API など、さまざまなデータを型安全に取り扱える Astro の機能です。

```ts:src/content.config.ts
import { defineCollection } from 'astro:content';
import { z } from 'astro/zod';

const blog = defineCollection({
  loader: async () => {
    // 外部 API から概要＋詳細データをまとめて取得する
    const items = await (await fetch(/* 外部 API */)).json();

    return items.map((item) => ({ id: item.id, ...item }));
  },
  schema: z.object({
    id: z.string(),
    title: z.string(),
    // ...
  }),
});

export const collections = { blog };
```

```ts
// 参照
const blog = (await getCollection('blog')).map((entry) => entry.data);
```

もともとはローカルの Markdown や MDX などを取り扱う機能でしたが、v5 で導入された [Content Layer API](https://astro.build/blog/content-layer-deep-dive/) により、外部 API も collection として扱えるようになりました。

loader で取得したデータは Content Layer のデータストアに保持されます。
各ページのビルド時には `getCollection()` 経由で同じスナップショットを参照できるため、API アクセス数を減らせます。

## 結果

API の N+1 を解消したことで DB への負荷が下がり、これまで完走できなかった Cloud Build 上のビルドが安定して完走するようになりました。

加えて、ビルド時間も大幅に短縮できました。
Cloud Build 上では改善前のビルドが完走しないため計測できませんが、ローカル環境であれば改善前でもビルド自体は通っていたので、ローカルでの計測値で比較します。
最終的に 1448 ページのビルドにかかった時間は以下のとおりです。

| | ビルド時間 |
| --- | --- |
| 改善前 | 42s |
| 改善後 | 8s |

## 学び

SQL の N+1 は気を付けていましたが、API の N+1 はあまり意識できていませんでした。

MPA/SPA では一覧を表示後、必要な詳細データを個別に取得する設計もあります。
その場合、ユーザーの操作ごとにリクエストが分散されるため許容できる範囲に収まることも多いと考えていました。

しかし SSG では、全ページ分のデータをビルド時に一度に取得しに行きます。
そのため N+1 がページ数分のリクエストとなり、短時間でデータソースへ集中します。
その結果、負荷という形で一気に表面化することが分かりました。

ビルド時にまとめてデータを取得する SSG の性質上、データ取得の効率は通常よりもシビアに効いてくるという学びになりました。
